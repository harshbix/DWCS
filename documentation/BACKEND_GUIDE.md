# Backend Guide

✅ **Status**: Implemented

The EcoCollect Tanzania backend is a hybrid system utilizing **Supabase (PostgreSQL + PostgREST)** as the primary data layer and a custom **Express.js API** for specific, isolated business logic.

## Express.js Application (`/backend`)
- **Purpose:** Acts as a secure enclave for operations that cannot be trusted to the client or the Edge.
- **Key Responsibilities:**
  - Processing webhooks from payment gateways (GePG / Mobile Money).
  - Executing heavy cron jobs or complex data aggregations that exceed Supabase RPC limits.
- **Architecture:** Standard MVC/Service-Repository pattern.
  - `src/server.ts`: Entry point.
  - `src/routes/`: Route definitions (`/api/health`, `/api/payments`).
  - `src/middleware/`: Global error handlers and request validators.

## Supabase Primary Backend
- **PostgREST API:** Automatically generated by Supabase from the PostgreSQL schema. Handles 95% of frontend CRUD operations via `@supabase/supabase-js`.
- **RPC Functions:** Stored procedures in Postgres used to combine multiple queries into a single network request (e.g., `v1_get_citizen_dashboard` joins user profiles, recent bills, and next schedules).

## Future Migration Opportunities
- **Technical Debt:** The Express backend adds deployment complexity.
- **Recommendation:** Migrate webhooks and cron jobs to **Supabase Edge Functions** (Deno-based) to unify the entire backend inside the Supabase ecosystem, reducing latency and DevOps overhead.
